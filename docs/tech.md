Техническое решение проекта

1. Опишите основную техническую идею, конструктивные особенности и т.д., положенные в основу создания продукта.

Основная техническая идея продукта — построение образовательной веб-платформы как набора связанных сервисов с чётким разделением ролей: клиентское приложение (интерфейс и состояние сессии), серверное API (бизнес-логика, вызов внешних ИИ-сервисов и учёт потребления), единая база данных с авторизацией и при необходимости подписками в реальном времени. Пользователь получает доступ ко всем функциям через браузер без установки ПО; платная нагрузка (речь, генерация текста, озвучка, определение слов) идёт через единый контур учёта баланса и списания по факту использования.

Конструктивные особенности:

• Архитектура «фронтенд — бэкенд — БД — внешние API». Фронтенд: SPA на Next.js 14 (App Router), рендер на клиенте, маршрутизация по разделам (главная, караоке, словарь, собеседник, прогресс, баланс). Бэкенд: REST API на Node.js и Express, принимает запросы от фронтенда, проверяет JWT (Supabase Auth), при платных действиях проверяет баланс и списывает средства, обращается к внешним сервисам (LLM, TTS, STT, при необходимости yt-dlp на сервере) и пишет данные в Supabase. База данных: PostgreSQL в Supabase с таблицами пользователей, сессий, видео, словаря, завершений сценариев/дебатов, оценок речи, баланса и транзакций; при необходимости используется Realtime для обновлений без перезагрузки.

• Единый контур голосового и текстового взаимодействия с ИИ. Распознавание речи (STT) выполняется через внешний API (Whisper-совместимый), текст передаётся в LLM (OpenAI-совместимый провайдер, например AITUNNEL) в формате streaming; ответ может озвучиваться через TTS. Один и тот же контур используется для свободного диалога, ролевых сценариев и дебатов — различаются только системные промпты, структура контекста и постобработка (фиксация шагов, фидбек, оценка речи). Это упрощает поддержку и добавление новых режимов.

• Караоке как отдельный поток данных. Клиент воспроизводит YouTube-видео через IFrame API; тайминг субтитров приходит с бэкенда (загрузка и парсинг SRT через yt-dlp, сохранение сегментов в БД). Синхронизация «текущая строка — время воспроизведения» выполняется на клиенте по сохранённым меткам времени; добавление слов/идиом/фразовых глаголов в словарь идёт через те же API словаря, что и в других разделах приложения, что обеспечивает единый словарь и повторения (SRS).

• Словарь и монетизация. Слова, идиомы и фразовые глаголы хранятся в Supabase с привязкой к пользователю; определения слов при промахе кэша запрашиваются через LLM и кэшируются в общей таблице. Повторения ведутся по алгоритму интервального повторения (SRS) с полями next_review_at и прогрессом в отдельных таблицах. Платные вызовы (LLM, TTS, STT, определение слов, анализ идиом/фразовых глаголов) на бэкенде проходят через общий слой: проверка JWT → проверка/списание баланса → вызов внешнего API → учёт usage и запись транзакции. Это даёт предсказуемую модель «оплата по факту использования» без подписок.

• Аналитика прогресса строится на уже накопленных данных: таблицы завершений ролевых сценариев и дебатов, таблица оценок речи (критерии, баллы), привязка оценок к завершениям по идентификаторам и времени. Расчёт средних баллов, трендов, слабых критериев и рекомендаций сценариев выполняется на фронтенде или через бэкенд по отфильтрованным выборкам за выбранный период; страница фокуса по сценарию/теме показывает все попытки и детализацию по критериям без дублирования логики хранения.

• Стандарты и совместимость. Взаимодействие фронтенда и бэкенда строится на REST; авторизация — на JWT (Supabase Auth). Вызовы к ИИ выполняются через OpenAI-совместимый API провайдера (например AITUNNEL), что позволяет при необходимости сменить модель или провайдера без переписывания бизнес-логики.

В итоге техническая идея сводится к тому, чтобы объединить разговорную практику с ИИ, караоке со словарём и аналитику прогресса в одной платформе с единой авторизацией, единым словарём и единым механизмом оплаты по факту использования, опираясь на современный веб-стек и внешние ИИ-сервисы без собственной инфраструктуры распознавания и синтеза речи и без собственных больших моделей.
