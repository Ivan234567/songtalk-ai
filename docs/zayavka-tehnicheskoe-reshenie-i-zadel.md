## 1. Техническое решение проекта

**Опишите основную техническую идею, конструктивные особенности и т.д., положенные в основу создания продукта.**

### Основная техническая идея

Продукт (платформа Speakeasy / SongTalk AI) — это образовательная веб-платформа для практики разговорного английского, построенная как набор связанных сервисов с чётким разделением ролей:

- **Клиентское приложение** — интерфейс, состояние сессии пользователя, воспроизведение медиа и отображение прогресса.
- **Серверное API** — бизнес-логика, вызов внешних ИИ-сервисов (распознавание и синтез речи, языковые модели), учёт потребления и списание с баланса.
- **Единая база данных** — хранение профилей, сессий, словаря, баланса, транзакций; при необходимости — подписки в реальном времени (Realtime) для обновлений без перезагрузки страницы.

Пользователь получает доступ ко всем функциям через браузер, без установки дополнительного ПО. Платная нагрузка (речь, генерация текста, озвучка, определение слов) проходит через единый контур: проверка баланса → списание по факту использования → запись транзакции. Таким образом, техническая идея сводится к объединению разговорной практики с ИИ, караоке со словарём и аналитики прогресса в одной платформе с единой авторизацией, единым словарём и единым механизмом оплаты по факту использования, без собственной инфраструктуры распознавания и синтеза речи и без собственных больших языковых моделей — с опорой на современный веб-стек и внешние ИИ-сервисы.

### Конструктивные особенности

**Архитектура «фронтенд — бэкенд — БД — внешние API».**

- **Фронтенд:** одностраничное приложение (SPA) на Next.js 14 с App Router, рендер на клиенте, маршрутизация по разделам: главная (дашборд), караоке, словарь, собеседник (ИИ-диалог, сценарии, дебаты), прогресс, пополнение баланса. Используются React-компоненты, единый Supabase-клиент для авторизации и запросов к данным, вызовы к бэкенду по REST для платных операций (чат, TTS, STT, словарь, оценка речи).
- **Бэкенд:** REST API на Node.js и Express. Принимает запросы от фронтенда, проверяет JWT (Supabase Auth), при платных действиях проверяет баланс и списывает средства, обращается к внешним сервисам (LLM, TTS, STT; при необходимости — yt-dlp на сервере для загрузки субтитров YouTube) и записывает данные в Supabase. Таймаут сервера увеличен для длительных ИИ-запросов (streaming).
- **База данных:** PostgreSQL в Supabase. Таблицы: пользователи (через Supabase Auth), сессии диалога, видео караоке, словарь (слова, идиомы, фразовые глаголы, кэш определений), завершения сценариев и дебатов, оценки речи, балансы пользователей (user_balances), транзакции (balance_transactions). При необходимости включена Realtime-репликация для выбранных таблиц — обновления приходят без перезагрузки страницы.

**Единый контур голосового и текстового взаимодействия с ИИ.**

Распознавание речи (STT) выполняется через внешний API, совместимый с Whisper; текст передаётся в языковую модель (LLM) по OpenAI-совместимому API провайдера (например AITUNNEL) в формате потоковой передачи (streaming); ответ при необходимости озвучивается через TTS. Один и тот же контур используется для свободного диалога, ролевых сценариев и дебатов — различаются только системные промпты, структура контекста (шаги сценария, позиция в дебатах) и постобработка: фиксация выполненных шагов, формирование фидбека, запрос оценки речи. Это упрощает поддержку и добавление новых режимов без дублирования кода.

**Караоке как отдельный поток данных.**

Клиент воспроизводит YouTube-видео через IFrame API; тайминг субтитров приходит с бэкенда (загрузка и парсинг SRT через yt-dlp, сохранение сегментов в БД). Синхронизация «текущая строка — время воспроизведения» выполняется на клиенте по сохранённым меткам времени. Добавление слов, идиом и фразовых глаголов в словарь идёт через те же API словаря, что и в разделах «Собеседник» и «Словарь», что обеспечивает единый словарь и единую систему повторений (SRS).

**Словарь и учёт потребления (монетизация).**

Слова, идиомы и фразовые глаголы хранятся в Supabase с привязкой к пользователю; определения слов при отсутствии в кэше запрашиваются через LLM и кэшируются в общей таблице. Повторения ведутся по алгоритму интервального повторения (SRS) с полями next_review_at и прогрессом в отдельных таблицах. Все платные вызовы (LLM, TTS, STT, определение слов, анализ идиом и фразовых глаголов) на бэкенде проходят через общий слой: проверка JWT → проверка и при необходимости списание баланса → вызов внешнего API → учёт потребления (usage) и запись транзакции. Это даёт предсказуемую модель «оплата по факту использования» без обязательной подписки.

**Аналитика прогресса.**

Строится на уже накопленных данных: таблицы завершений ролевых сценариев и дебатов, таблица оценок речи (критерии, баллы), привязка оценок к завершениям по идентификаторам и времени. Расчёт средних баллов, трендов, слабых критериев и рекомендаций сценариев выполняется на фронтенде или через бэкенд по отфильтрованным выборкам за выбранный период; страница фокуса по сценарию или теме дебата показывает все попытки и детализацию по критериям без дублирования логики хранения.

**Стандарты и совместимость.**

Взаимодействие фронтенда и бэкенда строится на REST; авторизация — на JWT (Supabase Auth). Вызовы к ИИ выполняются через OpenAI-совместимый API провайдера, что позволяет при необходимости сменить модель или провайдера без переписывания бизнес-логики. Единый helper для работы с backend JWT на фронтенде (проверка срока жизни токена, удаление просроченного, единые методы get/store/clear) обеспечивает синхронизацию баланса и доступа к платным функциям.

**Состояние функции оплаты.**

На данный момент пополнение баланса реализовано в виде ручного начисления администратором (для тестовых и ранних пользователей). Подключение платёжного шлюза ЮKassa для приёма оплаты банковской картой планируется после регистрации юридического лица (ИП или ООО): по требованиям ЮKassa для заключения договора и приёма платежей необходимы регистрация в качестве ИП или ООО, банковский счёт, оферта и политика конфиденциальности на сайте. Самозанятость для данного проекта не рассматривается. Интерфейс пополнения баланса (номиналы 300, 500, 1 000 ₽, ориентиры по минутам/озвучкам) и бэкенд-логика (topup_manual / topup_gateway) подготовлены к подключению шлюза после появления юридического лица.

## 2. Преимущества выбранного технического решения

**Опишите преимущества выбранного технического решения (простота конструкции/технологии, доступность/дешевизна материалов, улучшенные потребительские качества и т.д.)**

**Простота и унификация конструкции.**

Использование единого контура «STT → LLM → TTS» для всех режимов диалога (свободный разговор, сценарии, дебаты) снижает сложность поддержки и ускоряет добавление новых форматов практики за счёт смены промптов и контекста, а не переписывания пайплайна. Единый слой проверки баланса и списания на бэкенде исключает разрозненные правила тарификации и упрощает учёт. Словарь один на всё приложение (собеседник, караоке, отдельная вкладка) — пользователь не дублирует лексику, повторения (SRS) работают по единым правилам.

**Доступность и дешевизна технологий и инфраструктуры.**

Применяются массово распространённые, документированные технологии: Next.js, Node.js, Express, PostgreSQL (Supabase), REST, JWT. Не требуется развёртывание и сопровождение собственных моделей распознавания и синтеза речи или собственных больших языковых моделей — используются внешние API (OpenAI-совместимые провайдеры, Whisper-совместимый STT, TTS по символам). Инфраструктура строится на облачных сервисах (например Vercel — фронтенд, Railway — бэкенд, Supabase — БД и авторизация), что снижает стартовые затраты и упрощает масштабирование. Замена провайдера ИИ не требует смены архитектуры — достаточно смены ключей и базового URL.

**Улучшенные потребительские качества.**

Пользователь занимается через браузер с любого устройства, без установки приложений; один аккаунт даёт доступ ко всем разделам и единому словарю. Оплата только за фактическое использование (внутренний баланс, списание за запросы) снижает порог входа по сравнению с абонементом на занятия с репетитором и даёт прозрачность: при выборе суммы пополнения показываются ориентиры (например «~2 часа разговора с агентом», «до 800 озвучек»). Голосовой и текстовый ввод, озвучка ответов ИИ, подсказки ответа и встроенный переводчик снижают страх ошибки и повышают вовлечённость. Караоке с субтитрами и быстрым добавлением слов в словарь связывает практику восприятия на слух с накоплением лексики. Аналитика прогресса (баллы по критериям, тренды, рекомендации сценариев) даёт пользователю обратную связь и мотивацию.

**Масштабируемость и устойчивость.**

Разделение фронтенд / бэкенд / БД позволяет масштабировать компоненты независимо; использование облачных платформ с автоматическим деплоем ускоряет вывод обновлений. Единый механизм баланса и транзакций (включая атомарные RPC в БД) обеспечивает предсказуемость учёта и готовность к подключению платёжного шлюза (ЮKassa) после оформления ИП или ООО.

## 3. Имеющийся задел (в том числе научно-технический) для реализации проекта

**Опишите имеющиеся наработки в отношении создаваемого продукта, в т.ч. полученные в рамках реализации программы «УМНИК», объекты интеллектуальной собственности, которые могут быть использованы при создании продукта.**

**Реализованный функционал и кодовая база.**

Имеется работающий прототип веб-платформы (Speakeasy / SongTalk AI) с реализованными модулями:

- **Авторизация и профиль:** регистрация и вход через Supabase Auth, хранение профиля пользователя, единая работа с JWT на фронтенде (включая проверку срока жизни и синхронизацию с балансом).
- **ИИ-собеседник:** свободный диалог, ролевые сценарии (каталог и пользовательские, генерация сценариев через ИИ), дебаты (выбор темы, позиции, сложности); подсказка ответа, встроенный переводчик, настройки стиля общения; голосовой ввод (STT) и озвучка ответов (TTS); фидбек и оценка речи по критериям после завершения сценария или дебата.
- **Караоке:** загрузка субтитров по ссылке на YouTube (бэкенд, yt-dlp), воспроизведение с подсветкой текущей строки, добавление слов/идиом/фразовых глаголов в словарь, анализ идиом и фразовых глаголов по тексту песни, сохранение списка видео.
- **Словарь:** личная коллекция слов, идиом и фразовых глаголов с переводами, транскрипцией, контекстами; повторения по системе интервальных повторений (SRS); интеграция с собеседником и караоке; экспорт и импорт (CSV, JSON, Anki).
- **Прогресс:** аналитика по завершениям сценариев и дебатов, оценкам речи; средний балл, тренды, streak, рекомендации сценариев по слабым критериям; страницы фокуса по сценарию/теме с детализацией попыток.
- **Баланс и тарификация:** таблицы балансов и транзакций в Supabase, RPC для атомарного списания и пополнения; проверка баланса перед платными запросами, порог низкого баланса, история расходов и экспорт; тарифы по категориям (LLM, TTS, STT, словарь, оценка речи); интерфейс пополнения с номиналами и ориентирами по минутам/озвучкам; ручное пополнение администратором. Архитектура готова к подключению платёжного шлюза (ЮKassa) после регистрации ИП или ООО.

**Документация и проектные решения.**

Имеются внутренние документы: описание продукта и пользовательских сценариев, педагогическая и техническая модель учебной системы (Learning System Blueprint), контекст проекта и приоритеты (PROJECT_CONTEXT.md), техническое описание архитектуры и стека (tech.md), план монетизации и оферты (в т.ч. требования ЮKassa), материалы для заявок (описание продукта, цели SMART). Это формирует задел по продукту, педагогике и технике для дальнейшей реализации и масштабирования.

**Объекты интеллектуальной собственности.**

Права на созданный программный код, интерфейсы, тексты сценариев и описание учебной системы принадлежат разработчику/заявителю. При использовании в рамках проекта результатов интеллектуальной деятельности, права на которые принадлежат третьей организации (в т.ч. по программе «УМНИК»), в случае победы в конкурсе и при наличии гарантийного письма заявитель обязуется заключить лицензионный договор с этой организацией на использование соответствующего результата в деятельности стартапа. Конкретное описание таких результатов и условий лицензии прикладывается в соответствии с требованиями конкурса и гарантийного письма.
